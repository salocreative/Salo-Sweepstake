# Salo Sweepstake — FIFA World Cup 2026

A small, static site (no backend) that tracks our work sweepstake. Eight owners,
six teams each — see who's still alive, watch the group standings, and keep an
eye on the next fixtures.

Live data comes from [football-data.org](https://www.football-data.org/) (free
tier, competition code `WC`). A Node script pulls the latest data into JSON
snapshots in `./data`, and the site reads those snapshots directly. A GitHub
Actions workflow re-runs the script every hour and re-deploys to GitHub Pages.

## Project layout

```
.
├── index.html                # Single page, three tabs
├── assets/
│   ├── css/styles.css
│   └── js/app.js             # Vanilla JS, no build step
├── data/                     # JSON snapshots (regenerated by the build script)
│   ├── sweepstake.json       # Owners + their teams (from CSV)
│   ├── teams.json            # All WC teams from the API
│   ├── matches.json          # All WC fixtures + results
│   ├── standings.json        # Group standings
│   └── meta.json             # Build timestamp, competition info
├── scripts/
│   └── build-data.mjs        # Node script that talks to football-data.org
├── sweepstake_teams.csv      # Source of truth for who owns which team
└── .github/workflows/deploy.yml
```

## Local setup

1. Grab a free API key at
   <https://www.football-data.org/client/register>.

2. Run the data build once so the JSON files have real content:

   ```sh
   FOOTBALL_DATA_API_KEY=your_key_here npm run build:data
   ```

3. Serve the site (any static server will do):

   ```sh
   npm run serve   # python -m http.server on :8080
   # open http://localhost:8080
   ```

The build script logs any sweepstake teams it couldn't match against the API
(`No API match for ...`). If you see those, add the right spelling to the
`NAME_ALIASES` map at the top of `scripts/build-data.mjs`.

## Editing the sweepstake

Edit `sweepstake_teams.csv` (columns: `Owner,Team,Tier,Flag`) and re-run
`npm run build:data`. The frontend will pick up the new picks on next load.

Each owner has six picks, one per tier (Elite → Underdogs). With eight owners
that's the full 48-team World Cup field, one team per person per tier.

## Deploy to GitHub Pages

1. Create a new GitHub repo and push this directory.
2. In the repo, go to **Settings → Secrets and variables → Actions → New
   repository secret** and add:
   - `FOOTBALL_DATA_API_KEY` — your football-data.org token.
3. Go to **Settings → Pages** and set **Source** to **GitHub Actions**.
4. Trigger the workflow (push to `main`, or run *Build & deploy site* manually
   from the Actions tab).

The workflow:
- runs on every push to `main`, hourly (`cron: 0 * * * *`), and on manual
  dispatch,
- calls `node scripts/build-data.mjs` with the API key to refresh `./data`,
- uploads the whole directory as a Pages artifact and deploys it.

A `.nojekyll` file is included so GitHub Pages serves the `assets/` folder
unmodified.

## What's on the site

- **Leaderboard** — points per owner. 3 for a win, 1 for a draw, plus a 5-point
  bonus when any of your teams reaches the knockout rounds.
- **Teams tab** — each owner's 6 picks, grouped by tier (Elite, Strong,
  Challengers, Competitive, Outsiders, Underdogs) with their current W-D-L line
  and next fixture.
- **Groups tab** — all 12 World Cup groups, highlighting which owner owns each
  team.
- **Fixtures tab** — Upcoming / Live / Recent / All filters, and an owner
  filter so you can show only "Carl's fixtures this week".

## Notes / caveats

- football-data.org's free tier is limited to ~10 requests per minute. The
  build script makes three requests per run, so the hourly schedule sits well
  inside the budget.
- For knockout teams (e.g. "Winner G1"), the API uses placeholder names until
  the draw resolves; those fixtures will show as `TBD` on the site until the
  group stage finishes.
- Standings are only published while the tournament is active. Before kick-off
  the Groups tab will show a friendly empty state.
